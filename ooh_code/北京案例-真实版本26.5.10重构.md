最合理的方案不是“全区域随机生成 home locations”，而是：

> **以燕郊住宅小区 POI 作为 home-location anchors，再用 100 m 人口网格进行加权采样，最后在住宅 POI 周边生成乘客 home locations。**
> Meeting points 则优先使用燕郊真实公交站点、定制快巴站点、小区门口或主干路交叉口附近的真实点位。

这样最符合你的 **many-to-one DRT** 逻辑：燕郊是分散居住端，国贸是集中就业/换乘端。

## 1. 北京案例实验结构应这样设计

你的案例可以定义为：

> **Yanjiao–Guomao commuting corridor case**

其中：

| 元素                 | 设定                              |
| ------------------ | ------------------------------- |
| Origin side        | 燕郊居住区                           |
| Destination        | 国贸 / 大北窑 / 国贸地铁站附近              |
| Home locations     | 从燕郊住宅小区 POI 周边生成                |
| Meeting points     | 从真实公交站点、定制快巴站点、主干路交叉口中筛选        |
| Passenger arrivals | 主实验 90 人，敏感性分析 60/120/150 人     |
| Booking horizon    | 早高峰预约期，例如 6:30–8:30 或 7:00–9:00 |
| DRT service type   | many-to-one commuter DRT        |

这个设定有现实依据。北京市发改委文章明确提到，**818 路是北京和河北燕郊之间的大客流通勤线路，燕郊始发站为天洋城总站，北京终到站为郎家园站，是燕郊居民在国贸上班的主干线路之一**。([北京市发展和改革委员会][1]) 新京报也提到 818 路连接国贸和燕郊，是许多居住在燕郊的年轻人的重要通勤方式。([新京报][2])

## 2. Home locations 最合理的生成方法

我建议你采用这个三层生成逻辑：

### Step 1：划定燕郊住宅服务区

不要把 home 点生成在整条“燕郊—国贸”走廊上，而是只放在燕郊居住端。

建议范围：

> 燕顺路、神威北大街、学院街、迎宾路、行宫大街、天洋城、上上城、燕灵路口、兴达广场一带的居住片区。

论文中可以写：

> The origin-side service area is defined as the residential part of Yanjiao, which represents the commuter-origin side of the Yanjiao–Guomao corridor.

---

### Step 2：获取住宅 POI，作为 home anchors

住宅 POI 用高德地图最合适。高德 POI 2.0 支持关键字搜索、周边搜索、多边形区域搜索和 ID 搜索；多边形区域搜索可以用 polygon 限定研究区。([高德开放平台][3]) 高德官方文档还说明，POI 搜索可以返回 `name`、`location`、`type`、`typecode`、`address`、`adcode` 等字段。([高德开放平台][3])

你需要抓这些类型：

| 用途    | 高德 POI typecode |
| ----- | --------------: |
| 住宅小区  |          120302 |
| 宿舍    |          120303 |
| 社区中心  |          120304 |
| 住宅区大类 |          120300 |

高德 POI 分类表中，`120302` 对应“商务住宅；住宅区；住宅小区”，`120303` 对应“宿舍”，`120304` 对应“社区中心”。

建议你最终保留字段：

```text
poi_id, name, lon, lat, type, typecode, address, adcode
```

---

### Step 3：用人口网格给住宅 POI 加权

住宅 POI 只能告诉你“哪里有小区”，但不能告诉你“哪里人口更多”。所以最好叠加 100 m 人口网格。

最推荐的数据是：

> **A 100 m gridded population dataset of China’s seventh census**

这套数据使用中国第七次人口普查数据和地理空间大数据生成 100 m 中国人口网格。([ESSD][4]) Figshare 页面说明，该数据是基于 2020 年第七次人口普查、stacking ensemble learning 和 geospatial big data 生成的 100 m 人口网格。([Figshare][5])

如果这个数据下载或处理麻烦，备用数据可以用 WorldPop China 100 m。WorldPop 的 China 100m population 数据提供约 100 m 分辨率、WGS84 坐标系、单位为每个 grid square 的估计人口数。([WorldPop][6])

## 3. 具体生成算法

你可以这样生成 90 个 home locations：

### 方法 A：住宅 POI + 人口加权采样，最推荐

先得到燕郊住宅 POI 集合：

[
R={r_1,r_2,\ldots,r_N}
]

每个住宅 POI 有坐标：

[
(x_i,y_i)
]

然后从人口网格中计算每个住宅 POI 周边的人口权重，例如取 300 m buffer 内人口总和：

[
w_i=\sum_{g\in \mathcal{G}(r_i,300m)} Pop_g
]

再归一化：

[
p_i=\frac{w_i}{\sum_{j=1}^{N}w_j}
]

然后按照 (p_i) 抽取 90 个住宅 POI 作为 home anchors。每个被抽中的 POI 周围再加入小扰动：

[
h_n = r_i + \epsilon_n
]

其中：

[
\epsilon_n \sim \mathcal{N}(0,\sigma^2 I)
]

建议 (\sigma) 设置为 **50–100 m**，并限制 home location 不能偏离住宅 POI 超过 **150–200 m**。

这个方法最好，因为它满足三个条件：

1. home 点来自真实住宅小区；
2. 人口密集地区生成更多 home 点；
3. 点不是完全重叠在 POI 坐标上，而是分布在小区周边，更像真实住户位置。

## 4. Meeting points 应怎样生成

meeting points 不建议随机生成。最合理的是：

> **从真实公交站点和定制快巴站点中筛选 meeting point candidates。**

因为你的 DRT 是 many-to-one 通勤服务，meeting point 本质上应当是乘客可步行到达、车辆可停靠、容易组织上车的位置。

建议候选集合来自三类点：

| 类型          | 是否推荐 | 说明                     |
| ----------- | ---- | ---------------------- |
| 真实公交站点      | 强烈推荐 | 最符合接驳点/上车点逻辑           |
| 定制快巴站点      | 强烈推荐 | 与燕郊—国贸真实通勤场景完全一致       |
| 小区门口/主干路交叉口 | 推荐   | 可作为补充候选 meeting points |
| 任意随机点       | 不推荐  | 审稿人会质疑现实可停靠性           |

## 6. 如何拿到这些站点的经纬度

最推荐用高德公交信息查询或高德 POI 查询。

高德公交信息查询 API 明确支持公交站 ID 查询、公交站关键字查询、公交路线 ID 查询、公交路线关键字查询，返回 JSON 或 XML。([高德开放平台][11]) 公交站关键字查询接口为：

```text
https://restapi.amap.com/v3/bus/stopname?key=你的Key&city=110000&keywords=郎家园
```

高德文档说明，公交站关键字查询返回字段包括：

```text
id, name, location, adcode, citycode, buslines
```

其中 `location` 就是公交站经纬度。([高德开放平台][11])

但是注意：燕郊属于河北三河/廊坊，不一定用 `city=110000` 能全部查到。建议分别查：

```text
city=110000  # 北京侧
city=131000  # 廊坊侧，可尝试
```

如果公交信息查询权限不好申请，也可以直接用高德 POI 2.0 搜索公交站点。公交 POI 类型码可以用：

| 类型     | 高德 typecode |
| ------ | ----------: |
| 公交车站相关 |      150700 |
| 普通公交站  |      150702 |
| 旅游专线车站 |      150701 |
| 班车站    |      150800 |

高德分类表中，`150700` 是“交通设施服务；公交车站；公交车站相关”，`150702` 是“普通公交站”。

## 8. 论文中最稳妥的描述

你可以直接这样写：

> In the Beijing case study, we construct a many-to-one commuting scenario along the Yanjiao–Guomao corridor. The origin-side home locations are generated within the residential area of Yanjiao. Specifically, residential community POIs are first collected from public web-map services and used as candidate residential anchors. To reflect realistic spatial heterogeneity in residential demand, each residential anchor is weighted by the surrounding 100 m gridded population count derived from China’s seventh national population census. Passenger home locations are then sampled around these residential anchors with small spatial perturbations. Candidate meeting points are selected from real bus stops and customized commuter-bus stops in Yanjiao, including stops along the Yanjiao–Guomao customized commuter-bus service and the 818 cross-boundary bus corridor.

中文对应就是：

> 北京案例采用燕郊—国贸通勤走廊。home locations 在燕郊住宅区内生成，首先从真实住宅小区 POI 中提取候选住宅点，再结合 100 m 人口网格进行加权采样。meeting points 从真实公交站点和京津冀定制快巴站点中筛选，以保证候选上车点具备现实可达性和可停靠性。

## 10. 最关键结论

你应该采用：

> **住宅 POI + 人口网格加权生成 home locations；真实公交站/定制快巴站生成 meeting points；终点固定为国贸。**

不要采用：

> **在整个服务区域内均匀随机生成 home locations。**

因为后者虽然简单，但现实解释弱；前者能清楚说明：

> Home locations follow realistic residential spatial distribution, while meeting points are selected from real transit-accessible locations.

这对投稿最稳。
